# Team Setup Guide

DevEx excels at creating consistent development environments across team members, operating systems, and deployment environments. This guide covers best practices for implementing DevEx in team and organizational settings.

## Implementation Strategies

### Option 1: Shared Configuration Repository

Best for teams that want centralized control:

```bash
# Team members clone the shared config
git clone https://github.com/yourcompany/devex-config.git ~/.devex

# Install DevEx with team configuration
devex install

# Stay updated with team changes
cd ~/.devex && git pull
devex install --update
```

### Option 2: Configuration Templates

Best for teams that want flexibility with standards:

```bash
# Create team template
devex template create --name company-standard \
  --from-config configs/shared/applications.yaml

# Team members use template
devex template apply company-standard

# Customize per developer needs
devex install additional-tools --category optional
```

### Option 3: Layered Configuration

Best for complex organizations with multiple teams:

```yaml
# ~/.devex/config-layers.yaml
layers:
  - name: "company-base"
    source: "https://github.com/company/devex-base"
    priority: 10
    
  - name: "team-frontend"
    source: "https://github.com/company/devex-frontend"
    priority: 20
    condition: "team == 'frontend'"
    
  - name: "project-mobile"
    source: "https://github.com/company/devex-mobile"
    priority: 30
    condition: "project == 'mobile-app'"
    
  - name: "personal"
    source: "~/.devex/personal.yaml"
    priority: 100
```

## CI/CD Integration

### GitHub Actions Workflow

```yaml
# .github/workflows/devex-validation.yml
name: Validate DevEx Configuration

on:
  pull_request:
    paths:
      - 'configs/**'
      - '.devex/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Install DevEx
        run: |
          # Install DevEx with version pinning for CI reproducibility
          curl -fsSL https://devex.sh/install | bash
        
      - name: Validate Configuration
        run: |
          echo "$HOME/.local/share/devex/bin" >> $GITHUB_PATH
          devex config validate configs/shared/applications.yaml
          devex config validate configs/roles/
          
      - name: Test Installation (Dry Run)
        run: |
          devex install --config configs/shared/applications.yaml --dry-run
```

### Docker Development Environment

```dockerfile
# Dockerfile.dev
FROM ubuntu:22.04

# Install DevEx with security considerations
RUN apt-get update && apt-get install -y curl sudo ca-certificates
RUN curl -fsSL https://devex.sh/install | bash

# Copy team configuration
COPY configs/ /opt/devex-config/

# Install team tools
RUN /root/.local/share/devex/bin/devex install --config /opt/devex-config/shared/applications.yaml

# Set up development user
RUN useradd -m -s /bin/bash developer
USER developer
WORKDIR /home/developer

# Apply user configuration
RUN /root/.local/share/devex/bin/devex config apply --user
```

## Team Policies and Standards

### Configuration Standards

<Callout type="info">
**Team Configuration Standards:**

1. **Version Pinning**: Pin language versions for consistency
2. **Security Tools**: Include security scanners and linters
3. **Documentation Tools**: Standardize on documentation formats
4. **Testing Tools**: Include testing frameworks and tools
5. **Code Quality**: Enforce code formatting and linting
</Callout>

Example standards file:

```yaml
# configs/standards.yaml
standards:
  programming_languages:
    node:
      version: "18.17.0"  # LTS version
      package_manager: "pnpm"
      
    python:
      version: "3.11.5"
      package_manager: "pip"
      virtual_env: "venv"
      
  code_quality:
    linters:
      - eslint
      - pylint
      - golangci-lint
      
    formatters:
      - prettier
      - black
      - gofmt
      
  security:
    scanners:
      - snyk
      - bandit
      - gosec
      
  testing:
    frameworks:
      - jest
      - pytest
      - go-test
```

### Role-Based Access Control

Control access to different tools based on team roles:

```yaml
# configs/access-control.yaml
roles:
  junior_developer:
    allowed_categories:
      - development
      - editors

    allowed_categories: ["all"]
    
  devops_engineer:
    allowed_categories: ["all"]
    required_tools:
      - kubectl
      - terraform
      - ansible
```

## Troubleshooting Team Issues

### Common Team Challenges

<Tabs items={['Version Conflicts', 'Platform Differences', 'Network Issues', 'Permission Problems']}>
  <Tab value="Version Conflicts">
    **Problem**: Team members have different tool versions
    
    **Solution**:
    ```bash
    # Pin versions in team configuration
    applications:
      development:
        - name: node
          install_method: mise
          install_command: node@18.17.0  # Specific version
          global: true
    
    # Verify team alignment
    devex report versions --team
    ```
  </Tab>
  
  <Tab value="Platform Differences">
    **Problem**: Tools work differently across Linux/macOS/Windows
    
    **Solution**:
    ```yaml
    applications:
      development:
        - name: universal-tool
          linux:
            install_method: apt
            install_command: tool-name
          macos:
            install_method: brew
            install_command: tool-name
          windows:
            install_method: winget
            install_command: Publisher.ToolName
    ```
  </Tab>
  
  <Tab value="Network Issues">
    **Problem**: Corporate firewall blocks downloads
    
    **Solution**:
    ```bash
    # Configure proxy
    export https_proxy=http://proxy.company.com:8080
    devex config set proxy.https "$https_proxy"
    
    # Use internal mirrors
    devex config set mirrors.npm "https://npm.company.com"
    devex config set mirrors.python "https://pypi.company.com"
    ```
  </Tab>
  
  <Tab value="Permission Problems">
    **Problem**: Insufficient permissions for system packages
    
    **Solution**:
    ```bash
    # Use user-space installations
    devex config set install.prefer_user_space true
    
    # Or request admin privileges
    devex install --request-admin --reason "Team development setup"
    ```
  </Tab>
</Tabs>

## Best Practices for Teams

<Callout type="tip">
**Team DevEx Best Practices:**

1. **Start Simple**: Begin with core tools, add complexity gradually
2. **Version Everything**: Use version control for all configurations
3. **Test Changes**: Validate configuration changes before rolling out
4. **Document Decisions**: Record why specific tools were chosen
5. **Regular Reviews**: Periodically review and update team standards
6. **Feedback Loop**: Collect feedback from team members regularly
7. **Security First**: Include security tools and practices by default
8. **Backup Configurations**: Keep backups of working configurations
</Callout>

### Team Workflow Example

```bash
# 1. Developer proposes new tool
git checkout -b add-new-tool
# Edit configs/shared/applications.yaml
git commit -m "Add new-tool for improved productivity"

# 2. Team reviews via PR
# - Automated validation runs
# - Team members review and test
# - Security team approves

# 3. Merge and rollout
git checkout main
git merge add-new-tool

# 4. Team updates their environments
# Automated notification sent to team
# Team members run: devex team sync
```
